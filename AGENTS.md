# PRIORITY RULE - 前端修改安全 (MANDATORY)

修改前端代码时，必须先读取目标文件及相关格式化配置，确认现有编码、引号风格和实现方式后，再进行最小化修改。

Rules:
- 禁止整文件重写、批量替换或顺手重构无关代码
- 必须保持原文件编码、换行风格、引号风格和现有代码风格
- 遇到中文文案、注释或疑似非 UTF-8 文件时，修改后必须检查是否出现乱码或异常字符
- 禁止基于猜测直接修改，必须先通过代码和报错信息定位真实原因
- 修改后必须自检 diff，确认无乱码、无引号错误、无 JSX/TSX 语法错误、无无关改动
- 如可执行，必须运行 `lint`、`build`、类型检查或相关测试进行验证
- 如果发现编码异常、文件已存在乱码、或当前风格无法判断，必须先停止并说明风险

# PRIORITY - 先验证再行动（强制）

**遇到问题或需求时，禁止先入为主假设原因；须先分析验证、精准定位，再针对性解决。**

## 规则

- **禁止假设驱动**：不要基于猜测直接写代码或执行操作。
- **先诊断后治疗**：先收集事实、验证现状，再谈原因与方案。
- **精准定位**：通过日志、查询、API 测试等确认真实根因。
- **最小化行动**：确认根因后，采取最小、最精准的修复，避免大范围改动。
- **验证闭环**：修复后立即验证，确认问题已解决。

## 推荐流程

1. 收集信息 → 读代码/日志/数据，弄清真实状态
2. 分析验证 → 用事实确认根因，非假设
3. 制定方案 → 基于根因做最小化修复方案
4. 执行修复 → 针对性实施
5. 验证结果 → 确认有效
